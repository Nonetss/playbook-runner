## Context

The backend serves an OpenAPI reference (Scalar UI at `/scalar`, spec at `/openapi.json`) generated by `@orpc/openapi`'s `OpenAPIHandler` over `appRouter` (`apps/backend/src/routers/docs.ts`). Everything the reference shows comes from each procedure's builder chain: `.route({ summary, description, tags, method, ... })`, `.input(zodSchema)`, `.output(zodSchema)`, and `.errors({...})`.

Current inventory (per `packages/api/src/routers/*.ts`):

- All ~47 procedures have `summary`, `tags`, `method`.
- `description` is missing on several (`jobs.ts`, some `inventory.ts`).
- Typed `.output()` exists only on `run.*` (4 procedures) and one in `config`.
- `.errors()` exists only on `config.apiKeys.list`, which is a hand-rolled "test" carrying a full 5-code error map plus noise fields (`deprecated: false`, `inputStructure`/`outputStructure: "compact"`, `successStatus: 200`).
- Base procedures (`packages/api/src/index.ts`): `publicProcedure = o`, `protectedProcedure = publicProcedure.use(requireAuth)` — no shared `.errors()`.

Handlers return Drizzle-inferred rows (e.g. `Credential`, `Job`, `Playbook`) or small ad-hoc shapes. Dates come back as `Date`; `run.*` already models these with `z.coerce.date()`.

## Goals / Non-Goals

**Goals:**

- One enforced documentation shape for every endpoint: metadata + typed `.output()` + reachable `.errors()`.
- Zero-duplication error handling via common errors on the base procedures.
- Reusable per-resource response schemas feeding `.output()` and exported as types.
- A written standard so new endpoints comply by default.

**Non-Goals:**

- Changing any endpoint's runtime behavior, path, or wire response.
- Reworking auth, the RPC transport, or the Scalar setup itself.
- Adding new endpoints (the `scripts`/`command` features already landed).
- Runtime response validation as a new guarantee (schemas document; handlers already return typed rows — we validate at the edges we already do, not add global enforcement).

## Decisions

### Decision 1: Common errors on the base procedures

Declare shared errors once in `packages/api/src/index.ts`:

```ts
export const publicProcedure = o.errors({
  INTERNAL_SERVER_ERROR: { message: "Internal server error", status: 500 },
})
export const protectedProcedure = publicProcedure
  .errors({
    UNAUTHORIZED: { message: "Authentication required", status: 401 },
    FORBIDDEN: { message: "Not allowed", status: 403 },
  })
  .use(requireAuth)
```

Every endpoint inherits these; endpoints add only their specifics (`NOT_FOUND`, `BAD_REQUEST`, `CONFLICT`, `PRECONDITION_FAILED`). The `requireAuth` middleware's `throw new ORPCError("UNAUTHORIZED")` now aligns with a declared error.

- *Rationale:* removes ~20 lines of boilerplate per endpoint; single source of truth; matches the chosen approach.
- *Alternative rejected:* a `commonErrors` constant spread into each `.errors({ ...commonErrors })` — explicit but must be remembered on all ~47 endpoints and re-lists them in the per-endpoint OpenAPI noise.

### Decision 2: Reusable response schemas per resource

For each resource, define Zod response schemas colocated with its router (e.g. `credentialSchema`, `deviceSchema`, `jobSchema`, `playbookSchema`, `scriptSchema`) and export inferred types. `list` endpoints use `z.array(schema)`; `get`/mutations use the schema (or `schema.nullable()` where the handler can return null). Dates use `z.coerce.date()` to match `run.*`. Where a handler currently returns `X | null`, either keep `.nullable()` in the schema or (preferred) declare `NOT_FOUND` and narrow the output — decided per endpoint during apply.

- *Rationale:* one schema per resource avoids drift between `get` and `list`; exported types can replace scattered `Awaited<ReturnType<...>>` inference on the frontend later.
- *Note:* schemas are derived from the Drizzle row shape; keep them adjacent to the handler so they're updated together. A follow-up could generate them from Drizzle, out of scope here.

### Decision 3: Trim the reference example to the real standard

Normalize `config.apiKeys.list` down to the standard: keep `summary`/`description`/`tags`/`method`, keep `.output()` and a *reachable* `.errors()` (drop `NOT_FOUND`/`BAD_REQUEST` if `list` cannot raise them), and remove `deprecated: false`, `inputStructure`/`outputStructure: "compact"`, and `successStatus: 200` (all framework defaults). `successDescription` is optional — keep only where it adds signal.

- *Rationale:* the "most complete" test is actually over-documented with defaults; the standard should be complete *and* clean.

### Decision 4: Roll out router-by-router, mechanical and reviewable

Apply in dependency-free order, one router per task, verifying `check-types` and a regenerated `/openapi.json` after each. Order: base procedures first (unlocks inherited errors), then `config` (normalize the example), then `credentials`, `playbooks`, `scripts`, `inventory`, `jobs`, `run` (already has outputs — add errors + descriptions), then root `healthCheck`/`privateData`.

- *Rationale:* small, independently verifiable steps; the OpenAPI diff is the acceptance check.

## Risks / Trade-offs

- **`.output()` schema drifts from the real handler return** → Mitigation: colocate schemas with handlers; rely on `check-types` (oRPC infers handler return against `.output()`), and spot-check `/openapi.json`.
- **Nullable vs NOT_FOUND ambiguity** (`get` returns `X | null` today) → Mitigation: decide per endpoint at apply time — prefer declaring `NOT_FOUND` and returning non-null, fall back to `.nullable()` only where null is a valid documented response, to avoid changing client-visible behavior.
- **Declaring errors an endpoint can't actually raise** → Mitigation: the spec forbids it; add only codes the handler (or its resolver) can throw.
- **Large mechanical diff across all routers** → Mitigation: router-per-task rollout, no behavior change, OpenAPI diff reviewed per step.
- **Base-procedure `.errors()` changes error typing for handlers** → Mitigation: additive (declares, doesn't remove); existing `throw new ORPCError(...)` calls keep working and now map to declared codes.

## Migration Plan

Purely additive documentation/typing. No DB, no config, no runtime behavior change. Land base-procedure errors first, then routers incrementally; each step is independently deployable and the generated `/openapi.json` is the acceptance artifact. Rollback = revert; nothing stateful is touched.

## Open Questions

- For `get`-by-id endpoints that currently return `null` when absent: standardize on `NOT_FOUND` (behavior-preserving on the wire only if clients don't rely on `null`)? (Proposed: declare `NOT_FOUND` where the frontend doesn't depend on a `null` body; otherwise document `.nullable()`.)
- Where should the written standard live — a `packages/api/README.md`, an `AGENTS.md`/`CLAUDE.md` section, or a doc comment on the base procedures? (Proposed: `packages/api/README.md` plus a pointer comment in `index.ts`.)
- Do we also want to export the new response schemas for the frontend to consume as its resource types now, or leave that to a follow-up? (Proposed: export them; adopt on the frontend later.)
